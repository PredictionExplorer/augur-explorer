# ADR-0005: OpenAPI-first API v2

Date: 2026-07-10. Status: accepted.

## Context

The v1 API is a compatibility contract around a historical wire format:
operation-shaped paths, pagination in path segments, mixed field naming,
floating-point ETH amounts, and a `{"status":1,"error":""}` envelope. The
parity suite intentionally freezes those responses while clients migrate, so
cleaning up v1 in place would create avoidable breakage.

The stdlib router, pgx repositories, injected contract-state component, and
shared test fixtures now provide the foundation for a separately versioned
API. V2 needs one contract source, compile-time request/response types, stable
pagination under concurrent inserts, and an explicit retirement policy for
v1.

## Decision

### Contract and implementation

- `docs/openapi-v2.yaml` is the source of truth for `/api/v2/*`.
- V2 uses OpenAPI 3.0.3 until a stable oapi-codegen release supports OpenAPI
  3.1. Generated models, strict server interfaces, and Go 1.22+ `net/http`
  registration are committed and checked for drift in CI.
- Handlers live behind an injected `internal/api/v2.Server`. They do not read
  package-level mutable state.
- Paths are resource-oriented. The first vertical slice is the round-bid
  collection and item:
  - `GET /api/v2/cosmicgame/rounds/{round}/bids`
  - `GET /api/v2/cosmicgame/rounds/{round}/bids/{position}`
- Top-level success responses contain typed `data` and, for collections,
  `meta`; they do not carry the v1 status/error envelope.
- JSON fields use camelCase. Timestamps are RFC 3339 in UTC. Exact on-chain
  amounts are decimal strings whose unit is explicit in the field name
  (`*Wei`); binary floating-point amount fields are not exposed.
- Errors use RFC 9457 `application/problem+json`. Client errors may explain
  the rejected parameter, while 5xx responses never expose database, RPC, or
  internal implementation details.

### Current live state

- `GET /api/v2/cosmicgame/rounds/current` reads exactly one value copy from
  the injected contract-state cache, then combines it with the current
  round's database aggregates. It does not perform request-time RPC fallback
  reads, so one response cannot mix cache generations.
- The historical `PrizeClaimTimestamp` cache field is a duration returned by
  `getDurationUntilMainPrize()`, not a Unix timestamp. V2 exposes it as
  `secondsUntilMainPrize`; exact on-chain amounts and the main-prize time
  increment remain decimal strings with explicit units.
- An uninitialized cache or a documented contract-read sentinel means live
  state is temporarily unavailable. V2 returns a 503 RFC 9457 problem with
  `Retry-After` set to the five-second refresh interval. This includes a zero
  last-bidder address when the authoritative database count proves the round
  already has bids: the address field has not completed a successful live
  refresh. Invalid non-sentinel data and repository failures are internal 500
  responses. Neither response exposes RPC, database, or cache implementation
  details.

### Round prize taxonomy

- `GET /api/v2/cosmicgame/rounds/{round}/prizes` is a completed-round-only
  collection over the `cg_prize` registry. An open or missing round is 404;
  an exhausted page of an existing completed round is an empty 200 page.
- The database `ptype` values are a stable public enum: 0–2 main-prize
  ETH/CST/NFT; 3–4 last-CST-bidder NFT/CST; 5–6 endurance-champion NFT/CST;
  7–9 chrono-warrior ETH/CST/NFT; 10–12 bidder-raffle ETH/CST/NFT; 13–14
  RandomWalk-staker raffle CST/NFT; and 15 the round-wide Cosmic Signature
  staking ETH allocation.
- Each v2 prize exposes only the asset field implied by that enum
  (`ethAmountWei`, `cstAmountWei`, or `nftTokenId`). The staking allocation
  deliberately omits `winnerAddress`; its prize type identifies the
  beneficiary set. Legacy float amounts, internal address IDs, hardcoded
  claim flags and empty token metadata are not part of the contract.

### Round raffle records

- `/rounds/{round}/prizes` describes awarded assets. The raffle sub-resources
  expose the underlying payout records needed for withdrawal state and the
  combined NFT+CST award:
  - `raffle-eth-deposits` is the bidder-raffle PrizesWallet credit collection
    (`cg_prize_deposit` joined to `cg_prize` type 10); it excludes chrono-
    warrior and global/user wallet histories.
  - `raffle-nft-winners` reads `cg_raffle_nft_prize`; required `pool=bidder`
    or `pool=randomWalkStaker` replaces v1's two arrays in one mega-response.
- Both collections are completed-round-only. Open or missing rounds are 404;
  an exhausted page of an existing round is an empty 200 page.
- V2 orders ETH deposits by `(winner_index, evtlog_id)` and NFT winners by
  `(winner_idx, evtlog_id)` within a pool. The latter intentionally replaces
  v1's database-row-ID order with an immutable public keyset.
- ETH/CST values remain exact decimal wei strings. Internal record/address
  IDs and legacy floating-point amount fields are not exposed.

### Pagination

- List endpoints use `limit` plus an opaque `cursor`; v2 does not expose
  offsets.
- Cursors are bounded, base64url-encoded, versioned payloads. A cursor is
  scoped to its endpoint inputs and is rejected if reused for another round.
- Round bids are ordered by the immutable composite
  `(bid_position, evtlog_id)`. The cursor carries both values, preventing
  gaps or duplicates even if corrupt historical data repeats a bid position.
- Round prizes are ordered by the `cg_prize` primary-key suffix
  `(ptype, winner_index)` within one round. Both values are carried in the
  round-scoped cursor and backed by a matching read index.
- Raffle ETH cursors carry `(winner_index, evtlog_id)`. Raffle NFT cursors
  additionally carry the requested pool, so reuse across bidder and
  RandomWalk-staker collections is rejected.
- The default limit is 50 and the maximum is 200. `nextCursor` is omitted
  when the page is exhausted. Empty collections encode as `[]`, never `null`.
- Cursors are opaque rather than secret. They are strictly validated but not
  signed because changing one only changes a public read position.

### V1 migration and sunset

- V1 remains byte-identical while consumers migrate endpoint group by
  endpoint group. V2 additions must not regenerate v1 parity goldens.
- A v1 endpoint can be removed only after:
  1. every known first-party consumer has migrated,
  2. production route metrics show 30 consecutive days of zero v1 traffic,
     excluding documented synthetic probes, and
  3. a previously announced not-before date has passed.
- Deprecation and Sunset headers will be added only after the complete v1-to-v2
  mapping and rollout dates are published.

## Consequences

V2 can improve its contract without weakening the v1 freeze. Generated strict
interfaces make spec drift a compile failure, while response-validation and
route-drift tests catch semantic drift. Cursor pagination requires dedicated
keyset queries and indexes, but avoids the instability and increasing scan
cost of large offsets.

During migration both API versions are served by the same process. The v1
package globals remain compatibility debt until their endpoint groups are
retired; no new v2 code may depend on them.
