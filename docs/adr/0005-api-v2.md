# ADR-0005: OpenAPI-first API v2

Date: 2026-07-10. Status: accepted. The v2 endpoint inventory completed
2026-07-16 and v1 was formally deprecated the same day (spec `deprecated`
flags + RFC 9745 `Deprecation`/`Link` headers; the RFC 8594 `Sunset` header
ships once the sunset gates below produce an announced date via
`V1_SUNSET_AT`).

## Context

The v1 API is a compatibility contract around a historical wire format:
operation-shaped paths, pagination in path segments, mixed field naming,
floating-point ETH amounts, and a `{"status":1,"error":""}` envelope. The
parity suite intentionally freezes those responses while clients migrate, so
cleaning up v1 in place would create avoidable breakage.

The stdlib router, pgx repositories, injected contract-state component, and
shared test fixtures now provide the foundation for a separately versioned
API. V2 needs one contract source, compile-time request/response types, stable
pagination under concurrent inserts, and an explicit retirement policy for
v1.

## Decision

### Contract and implementation

- `docs/openapi-v2.yaml` is the source of truth for `/api/v2/*`.
- V2 uses OpenAPI 3.0.3 until a stable oapi-codegen release supports OpenAPI
  3.1. Generated models, strict server interfaces, and Go 1.22+ `net/http`
  registration are committed and checked for drift in CI.
- Handlers live behind an injected `internal/api/v2.Server`. They do not read
  package-level mutable state.
- Paths are resource-oriented. The first vertical slice is the round-bid
  collection and item:
  - `GET /api/v2/cosmicgame/rounds/{round}/bids`
  - `GET /api/v2/cosmicgame/rounds/{round}/bids/{position}`
- Top-level success responses contain typed `data` and, for collections,
  `meta`; they do not carry the v1 status/error envelope.
- JSON fields use camelCase. Timestamps are RFC 3339 in UTC. Exact on-chain
  amounts are decimal strings whose unit is explicit in the field name
  (`*Wei`); binary floating-point amount fields are not exposed.
- Errors use RFC 9457 `application/problem+json`. Client errors may explain
  the rejected parameter, while 5xx responses never expose database, RPC, or
  internal implementation details.

### Current live state

- `GET /api/v2/cosmicgame/rounds/current` reads exactly one value copy from
  the injected contract-state cache, then combines it with the current
  round's database aggregates. It does not perform request-time RPC fallback
  reads, so one response cannot mix cache generations.
- The historical `PrizeClaimTimestamp` cache field is a duration returned by
  `getDurationUntilMainPrize()`, not a Unix timestamp. V2 exposes it as
  `secondsUntilMainPrize`; exact on-chain amounts and the main-prize time
  increment remain decimal strings with explicit units.
- An uninitialized cache or a documented contract-read sentinel means live
  state is temporarily unavailable. V2 returns a 503 RFC 9457 problem with
  `Retry-After` set to the five-second refresh interval. This includes a zero
  last-bidder address when the authoritative database count proves the round
  already has bids: the address field has not completed a successful live
  refresh. Invalid non-sentinel data and repository failures are internal 500
  responses. Neither response exposes RPC, database, or cache implementation
  details.

### Round prize taxonomy

- `GET /api/v2/cosmicgame/rounds/{round}/prizes` is a completed-round-only
  collection over the `cg_prize` registry. An open or missing round is 404;
  an exhausted page of an existing completed round is an empty 200 page.
- The database `ptype` values are a stable public enum: 0–2 main-prize
  ETH/CST/NFT; 3–4 last-CST-bidder NFT/CST; 5–6 endurance-champion NFT/CST;
  7–9 chrono-warrior ETH/CST/NFT; 10–12 bidder-raffle ETH/CST/NFT; 13–14
  RandomWalk-staker raffle CST/NFT; and 15 the round-wide Cosmic Signature
  staking ETH allocation.
- Each v2 prize exposes only the asset field implied by that enum
  (`ethAmountWei`, `cstAmountWei`, or `nftTokenId`). The staking allocation
  deliberately omits `winnerAddress`; its prize type identifies the
  beneficiary set. Legacy float amounts, internal address IDs, hardcoded
  claim flags and empty token metadata are not part of the contract.

### Round raffle records

- `/rounds/{round}/prizes` describes awarded assets. The raffle sub-resources
  expose the underlying payout records needed for withdrawal state and the
  combined NFT+CST award:
  - `raffle-eth-deposits` is the bidder-raffle PrizesWallet credit collection
    (`cg_prize_deposit` joined to `cg_prize` type 10); it excludes chrono-
    warrior and global/user wallet histories.
  - `raffle-nft-winners` reads `cg_raffle_nft_prize`; required `pool=bidder`
    or `pool=randomWalkStaker` replaces v1's two arrays in one mega-response.
- Both collections are completed-round-only. Open or missing rounds are 404;
  an exhausted page of an existing round is an empty 200 page.
- V2 orders ETH deposits by `(winner_index, evtlog_id)` and NFT winners by
  `(winner_idx, evtlog_id)` within a pool. The latter intentionally replaces
  v1's database-row-ID order with an immutable public keyset.
- ETH/CST values remain exact decimal wei strings. Internal record/address
  IDs and legacy floating-point amount fields are not exposed.

### Round donation records

- The three round-scoped donation collections expose the immutable events
  recorded while bidding, rather than the legacy operation-shaped views:
  - `eth-donations` combines plain and info-carrying direct ETH donations.
    A `kind` discriminator identifies the variant; `contractRecordId` and the
    contract-provided `data` string are present only for `withInfo`.
  - `erc20-donations` exposes one row per token donation. Its exact quantity
    is named `amountBaseUnits`, not wei, because arbitrary ERC-20 contracts
    need not use 18 decimals.
  - `nft-donations` exposes the PrizesWallet donation index, token contract,
    token ID and recorded token URI.
- Donations can be made before a round closes, so these collections do not
  use the completed-round gate required by prizes and raffles. Any
  nonnegative round may be queried; a round with no matching events returns
  an empty 200 page, consistent with the round-bids collection.
- Charity-wallet transfers, user-scoped histories, ERC-20 aggregates and
  donated-asset claim events are separate resources and are not folded into
  the round donation event collections. Internal row/address IDs, legacy
  floating-point values and prize-claim joins are not exposed.

### Global statistics, ROI and claims

- `GET /api/v2/cosmicgame/statistics` is the database-backed global aggregate
  that complements `/rounds/current`. It performs no request-time RPC and
  does not recreate v1's dashboard mega-response. Exact ETH/CST totals are
  decimal wei strings; arbitrary ERC-20 quantities use base-unit strings.
- `statistics/counters` exposes the canonical bid, completed-round and donated
  NFT record counts. `statistics/leaderboard/roi` replaces offsets with a
  cursor scoped to `sort` and `minBids`; every sort has an immutable bidder
  tie-breaker. ROI and win rate are PostgreSQL-numeric decimal strings rather
  than JSON floats, while signed net profit remains exact wei.
- `statistics/claims` is a cursor-paginated, newest-round-first summary. It
  deliberately omits v1's unbounded inline asset arrays. A round's bounded
  detail lives at `/rounds/{round}/claims`, with independently paginated claim
  transactions, attached tokens and unclaimed items.
- Round claim detail is completed-round-only: open and missing rounds are
  404; a completed round without claimable records is a 200 response with
  non-nil empty section arrays. Claim asset variants are `eth`, `erc721` and
  `erc20`; each exposes only the exact amount/token fields that apply.

### Participant directories

- Six bounded directories replace the unpaginated v1 `statistics/unique/*`
  arrays: bidders, winners, ETH donors, CST stakers, RandomWalk stakers and
  addresses currently staking both token families. V2 uses the resource path
  `/statistics/participants/*`; the v1 `randomwalk`/`rwalk` alias pair is not
  reproduced.
- Directory rows come from the trigger-maintained aggregate tables and expose
  canonical addresses, non-negative counts and exact decimal wei strings;
  winner counts and ETH totals are reconstructed from canonical prize events
  because the legacy `cg_winner` aggregate is replay-sensitive.
  Internal address IDs, duplicate winner structures, deprecated counters and
  floating-point ETH fields are omitted. A bidder whose aggregate `max_bid`
  is the historical `-1` no-ETH-bid sentinel has no `maxBidWei` field.
  Winner ETH fields explicitly distinguish the largest main-prize component,
  total ETH won and total ETH spent.
- Bidders, winners, donors, CST stakers, RandomWalk stakers and dual stakers
  are ordered respectively by bid count, prize count, donated wei, reward
  wei, current staked-token count and combined current staked-token count.
  Every primary key is descending; the internal address ID is an ascending,
  unexposed tie-breaker.

### Bidding analytics

- Five database-only resources replace the v1 bidding chart endpoints:
  `statistics/bidding/activity`, `frequency`, `type-ratio`,
  `top-active-periods` and `time-bounds`. Windowed resources use required
  `from`/`to` query timestamps instead of positional path segments; no
  request-time RPC or dashboard mega-response is involved.
- Every request is bounded before reaching PostgreSQL. Windows are positive,
  at most five years, and time-series requests may produce at most 2,000
  buckets. Intervals must be positive; top-period analysis accepts at most
  100 bidders and returns at most 2,000 periods, rejecting broader results
  with 400. Time-series queries filter through the `cg_bid(time_stamp)` read
  index and aggregate each bid to one `DATE_BIN` before joining the zero-fill
  series; every analytics query also has a five-second request deadline.
  Together these bounds prevent untrusted `generate_series`, response
  amplification and bucket-by-row range joins.
- Activity and frequency retain the established first-hour-after-round-open
  exclusion. One-hour and one-day frequency buckets align to UTC epoch
  boundaries; other widths and all type-ratio buckets anchor at `from`.
  A legacy terminal bucket whose start equals the exclusive `to` boundary is
  not exposed by v2.
- Bid-type composition exposes raw counts and decimal-string percentages
  rounded from integer ratios. Legacy floating-point percentage fields and
  internal address IDs are not part of the contract.
- Top active periods rank the lifetime top bidders, then segment only their
  bids in `[from,to)`. Responses use canonical addresses and deterministic
  ordering. Spike detection keeps the existing dynamic-threshold algorithm;
  `recentSpikeIndex` is omitted unless a spike starts in the 30 days ending
  at the server's current time.

### Contract configuration and cached live reads

- The v1 dashboard's remaining contract surface is decomposed into five
  resources: the DB-backed address registry, cached configuration, cached
  balances, cached bid prices and cached special-winner standings. V2 does
  not recreate a dashboard endpoint.
- V2 handlers never perform request-time RPC. Contract constants refresh
  every five minutes, round prices/balances every five seconds and special
  winners every 30 seconds. Required unavailable groups return RFC 9457 503
  with `Retry-After` matching the failed group's schedule.
- Constants, variables and balances are serialized where they share
  mechanics or address dependencies. Contract calls and balance reads are
  pinned to one latest block and have bounded deadlines. Configuration is
  published only when its constant and variable fields carry the same
  mechanics generation.
- V1 mechanics expose a fixed CST bid reward and CST auction divisor. V2
  mechanics expose a static reward multiplier, the computed reward for the
  next bid, a CST auction duration and its duration-change divisor. Auction
  start timestamps are normalized to elapsed seconds and progress is clamped
  to `0..duration`.
- CosmicGame and charity balances are exact wei strings. The cached charity
  address used for a balance is carried with that balance generation; an
  address change invalidates the old balance.
- Special-winner contract calls and their round number are block-pinned. The
  optional last-CST-bid event ID is selected deterministically at or before
  that source block; a missing/failed optional DB lookup omits only that
  field.
- The complete field-by-field dashboard replacement is published in
  [api-v2-migration.md](../api-v2-migration.md). Remaining user histories,
  RandomWalk, CosmicToken and marketing-wallet resources stay separate slices.

### User profile foundation

- `GET /api/v2/cosmicgame/users/{address}` replaces the aggregate portion of
  the v1 user mega-response with one collection-free profile. It exposes exact
  bidding, canonical prize/raffle, direct ETH donation, transfer and staking
  statistics without internal address IDs, floating-point ETH, magic
  sentinels or request-time RPC.
- Prize totals are reconstructed from canonical prize/event rows rather than
  the replay-sensitive `cg_winner` aggregate. Bid totals and maxima likewise
  come from canonical bid rows. The bounded query benchmarks in the same
  latency class as the participant directories.
- A valid but not-yet-indexed wallet is still a valid resource identity: the
  endpoint returns a checksummed address with a stable zero-activity `200`
  shape. Malformed addresses are `400`; corrupt repository data and database
  failures are opaque `500` problems.
- `GET /api/v2/cosmicgame/users/{address}/bids` is the first history
  sub-resource. It reuses the established typed `Bid` representation and
  pages newest-first on immutable event-log ID. Prize, donation, transfer,
  staking, token, marketing and live-balance histories remain dedicated
  future sub-resources rather than recreating the v1 unbounded response.

### Pagination

- List endpoints use `limit` plus an opaque `cursor`; v2 does not expose
  offsets.
- Cursors are bounded, base64url-encoded, versioned payloads. A cursor is
  scoped to its endpoint inputs and is rejected if reused for another round.
- Round bids are ordered by the immutable composite
  `(bid_position, evtlog_id)`. The cursor carries both values, preventing
  gaps or duplicates even if corrupt historical data repeats a bid position.
- Round prizes are ordered by the `cg_prize` primary-key suffix
  `(ptype, winner_index)` within one round. Both values are carried in the
  round-scoped cursor and backed by a matching read index.
- Raffle ETH cursors carry `(winner_index, evtlog_id)`. Raffle NFT cursors
  additionally carry the requested pool, so reuse across bidder and
  RandomWalk-staker collections is rejected.
- Round donation collections are ordered newest first by immutable
  `evtlog_id`. Their cursors carry the round, event-log ID and resource kind,
  so a cursor cannot be reused across ETH, ERC-20 and NFT collections.
- ROI cursors carry the selected sort, `minBids`, exact sort key and internal
  bidder tie-breaker; the tie-breaker is never exposed in a response.
  Claims-summary cursors carry `(round_num, evtlog_id)`. Round-claim cursors
  additionally carry their section, round and section-specific stable key.
- Participant cursors carry the directory kind, exact aggregate sort key and
  internal address-ID tie-breaker. Reusing one directory's cursor on another
  directory is rejected. Unlike immutable event collections, participant
  ranks can move when aggregates change between requests; clients de-duplicate
  addresses while traversing and restart at the head for a fresh ranking.
- User-bid cursors carry the normalized wallet address and final event-log ID.
  Reuse against another wallet is rejected; `(bidder_aid, evtlog_id DESC)` is
  backed by a matching read index.
- The default limit is 50 and the maximum is 200. `nextCursor` is omitted
  when the page is exhausted. Empty collections encode as `[]`, never `null`.
- A cursor is a stable boundary, not a live subscription. When an open
  collection receives a new event ahead of a previously returned cursor, that
  event is intentionally outside the older traversal; clients following live
  rounds poll the head page without a cursor and use cursors to walk older
  records.
- Cursors are opaque rather than secret. They are strictly validated but not
  signed because changing one only changes a public read position.

### V1 migration and sunset

- V1 remains byte-identical while consumers migrate endpoint group by
  endpoint group. V2 additions must not regenerate v1 parity goldens.
- A v1 endpoint can be removed only after:
  1. every known first-party consumer has migrated,
  2. production route metrics show 30 consecutive days of zero v1 traffic,
     excluding documented synthetic probes, and
  3. a previously announced not-before date has passed.
- Deprecation and Sunset headers will be added only after the complete v1-to-v2
  mapping and rollout dates are published.

## Consequences

V2 can improve its contract without weakening the v1 freeze. Generated strict
interfaces make spec drift a compile failure, while response-validation and
route-drift tests catch semantic drift. Cursor pagination requires dedicated
keyset queries and indexes, but avoids the instability and increasing scan
cost of large offsets.

During migration both API versions are served by the same process. The v1
package globals remain compatibility debt until their endpoint groups are
retired; no new v2 code may depend on them.
